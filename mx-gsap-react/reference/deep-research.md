# Architecting High-Performance GSAP Animations in Next.js App Router Environments

**Key Points:**
*   The integration of the GreenSock Animation Platform (GSAP) within the Next.js App Router architecture necessitates a structural paradigm shift from traditional React single-page applications. 
*   Research suggests that meticulously isolating `"use client"` boundaries effectively preserves the performance benefits of React Server Components (RSCs) while permitting the imperative DOM manipulations required by GSAP [cite: 1, 2].
*   It seems likely that standardizing on context-driven `TransitionProvider` patterns is currently the most robust strategy for achieving fluid route transitions, bypassing the App Router's inherent lack of native exit-animation lifecycles [cite: 3, 4].
*   The evidence leans toward strictly segregating CSS-based transitions—particularly those generated by Tailwind CSS utility classes—from GSAP-controlled properties to prevent severe browser painting conflicts [cite: 5, 6].
*   Employing SSR-safe CSS rules (e.g., `visibility: hidden`) rather than relying on JavaScript-based initializations is fundamentally required to prevent the Flash of Unstyled Content (FOUC) during server hydration [cite: 7, 8].

**Introduction**
The migration from the Next.js Pages Router to the App Router represents a fundamental shift in how React applications are rendered, moving from a primarily client-hydrated model to a server-first architecture. While this brings substantial benefits to performance and Search Engine Optimization (SEO), it introduces complex challenges for imperative animation libraries like GSAP. GSAP requires direct access to DOM nodes and browser lifecycle hooks, which are explicitly unavailable in React Server Components [cite: 2, 9]. 

**Scope of the Technical Guide**
This comprehensive guide establishes the academic and practical standards for implementing GSAP within modern Next.js environments. It systematically addresses five critical architectural pillars: the strategic placement of `"use client"` directives to maintain optimal payload sizes, the orchestration of synchronous page route transitions, the mitigation of conflicts between inline styles and utility-first CSS frameworks like Tailwind, the eradication of FOUC during server-side rendering, and the implementation of dynamic imports for heavy animation modules. By adhering to the decision trees and anti-rationalization rules provided herein, engineers can orchestrate highly performant, animation-rich applications without compromising the fundamental benefits of the Next.js App Router.

## 1. "use client" Boundary Patterns in Server-First Architectures

The Next.js App Router defaults to Server Components, which run exclusively on the server and do not ship JavaScript to the browser [cite: 10]. GSAP, however, is deeply reliant on browser-specific APIs (`window`, `document`, `requestAnimationFrame`) and React client hooks (`useRef`, `useEffect`, `useLayoutEffect`) to target and animate DOM nodes [cite: 7, 9]. Therefore, GSAP animations must be executed strictly within Client Components.

The architectural challenge lies in minimizing the "client boundary." If a developer declares `"use client"` at the top of a root `layout.tsx` or `page.tsx` file, every child component imported into that file becomes a Client Component, severely bloating the JavaScript bundle and disabling server-side data fetching and static metadata generation [cite: 1, 11].

### Technical Reference: The "Children as Props" Wrapper Pattern

To safely introduce GSAP into a server-rendered layout—such as instantiating a global `ScrollSmoother` or an authentication transition wrapper—developers must utilize the "children wrapper" pattern [cite: 1, 12]. 

React inherently allows Server Components to be passed as `children` to Client Components [cite: 1, 2]. By creating a discrete Client Component wrapper, the GSAP logic is isolated, and the Next.js compiler retains the ability to render the nested children on the server before passing them as serialized nodes to the client wrapper.

**Code Example: Isolating ScrollSmoother in a Client Component Wrapper**

```tsx
// components/SmootherWrapper.tsx
"use client";

import React, { useRef } from "react";
import { gsap } from "gsap";
import { ScrollSmoother } from "gsap/ScrollSmoother";
import { useGSAP } from "@gsap/react";

// Register plugins once in the client boundary
if (typeof window !== "undefined") {
  gsap.registerPlugin(ScrollSmoother, useGSAP);
}

interface SmootherWrapperProps {
  children: React.ReactNode;
}

export default function SmootherWrapper({ children }: SmootherWrapperProps) {
  const wrapperRef = useRef<HTMLDivElement>(null);
  const contentRef = useRef<HTMLDivElement>(null);

  useGSAP(() => {
    // Instantiate ScrollSmoother only on the client
    ScrollSmoother.create({
      wrapper: wrapperRef.current,
      content: contentRef.current,
      smooth: 1.5,
      effects: true,
    });
  }, { scope: wrapperRef });

  return (
    <div ref={wrapperRef} id="smooth-wrapper">
      <div ref={contentRef} id="smooth-content">
        {children}
      </div>
    </div>
  );
}
```

**Code Example: Consuming the Wrapper in a Server Component**

```tsx
// app/layout.tsx
// Notice the ABSENCE of "use client" here. This remains a Server Component.
import type { Metadata } from "next";
import SmootherWrapper from "@/components/SmootherWrapper";
import { GeistSans } from "geist/font/sans";
import "./globals.css";

// Metadata remains perfectly functional
export const metadata: Metadata = {
  title: "GSAP Next.js Architecture",
  description: "High performance server components with client animations.",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body className={GeistSans.className}>
        {/* 
          The SmootherWrapper is a Client Component.
          The {children} are Server Components rendered on the server 
          and passed into the client boundary seamlessly.
        */}
        <SmootherWrapper>
          {children}
        </SmootherWrapper>
      </body>
    </html>
  );
}
```

### Decision Tree: Where to Place the `"use client"` Directive

| Architectural Requirement | Optimal Placement Strategy | Justification |
| :--- | :--- | :--- |
| **Global Layout Animation** (e.g., Page wrappers, Smooth Scrolling) | Create a dedicated `<ClientWrapper>` component; import into `layout.tsx`. | Preserves `layout.tsx` as a Server Component for SEO and `<head>` metadata injection [cite: 1]. |
| **Leaf Component Animation** (e.g., A single rotating logo or button) | Add `"use client"` directly to the specific `Logo.tsx` or `Button.tsx` file. | Isolates JS payload strictly to the element requiring interactivity, leaving the parent page as a Server Component. |
| **Page-Level Entrance Animation** (e.g., Entire page fades in) | Create a `<PageTransitionWrapper>` Client Component to wrap the `page.tsx` return statement. | Ensures server-side data fetching can still occur natively inside the `page.tsx` async component. |

### Anti-Rationalization Rules for "use client" Boundaries

**Rule 1: The Layout Conversion Fallacy**
*   **The Temptation:** When an engineer attempts to add `ScrollSmoother` or global page transitions to `layout.tsx`, the compiler throws a hook-related error. The engineer rationalizes, *"I will just add `'use client'` to the top of `layout.tsx` to fix the error."*
*   **Why it Fails:** Converting the root layout to a Client Component immediately breaks Next.js static metadata generation (`export const metadata`) [cite: 1]. It also forces every sub-route, sub-page, and component into the client bundle, disastrously impacting Total Blocking Time (TBT) and time-to-interactive metrics.

**Rule 2: The Direct Import Misunderstanding**
*   **The Temptation:** The engineer rationalizes, *"If I import a Server Component into my Client Component, it stays a Server Component."*
*   **Why it Fails:** In Next.js, importing a component directly into a file marked with `"use client"` implicitly forces that imported component to become part of the client bundle [cite: 10]. The only way to maintain the Server Component nature of a nested element is to pass it as a `children` prop from a higher-level Server Component [cite: 1, 2].

## 2. Route Transition Animations

Unlike the legacy Pages Router, which natively supported route transition interceptors (such as `react-transition-group` or Framer Motion's `AnimatePresence` within `_app.js`), the Next.js App Router poses severe limitations on exit animations. As of recent Next.js versions, when a user navigates to a new route, the current component tree is aggressively unmounted and replaced [cite: 13]. 

If an engineer attempts to trigger a GSAP exit animation upon a route change, the DOM elements are destroyed before the animation can calculate its first frame [cite: 3, 14]. To achieve seamless page transitions, we must invert the navigation lifecycle: intercept the navigation event, execute the GSAP exit animation, and only instruct Next.js to navigate (`router.push`) after the animation resolves [cite: 3, 4, 15].

### Technical Reference: The `TransitionProvider` and `TransitionLink` Pattern

This architectural pattern requires three elements:
1.  A globally persistent **Transition Context** to hold the animation state.
2.  A **Transition Provider** that renders an overlay (or controls page opacity) and executes the imperative GSAP `navigate()` function [cite: 4].
3.  A custom **TransitionLink** component that replaces the native `next/link`.

**Code Example: Transition Context and Provider**

```tsx
// context/TransitionContext.tsx
"use client";

import React, { createContext, useRef, useLayoutEffect } from "react";
import { useRouter, usePathname } from "next/navigation";
import { gsap } from "gsap";

export const TransitionContext = createContext<{ navigate: (href: string) => void }>({ 
  navigate: () => {} 
});

export const TransitionProvider = ({ children }: { children: React.ReactNode }) => {
  const router = useRouter();
  const pathname = usePathname();
  const overlayRef = useRef<HTMLDivElement>(null);
  const isAnimating = useRef(false);

  // Entrance Animation: Fires when the pathname changes (new route loads)
  useLayoutEffect(() => {
    if (overlayRef.current) {
      // Animate overlay off-screen to reveal new page
      gsap.to(overlayRef.current, {
        yPercent: -100,
        duration: 0.8,
        ease: "power3.inOut",
        onComplete: () => {
          isAnimating.current = false;
        }
      });
    }
  }, [pathname]); // usePathname acts as the trigger for entrance [cite: 4, 15]

  // Exit Animation & Programmatic Navigation
  const navigate = (href: string) => {
    if (isAnimating.current || pathname === href) return; [cite: 15]
    isAnimating.current = true;

    // Reset overlay position below screen, then animate it up
    gsap.set(overlayRef.current, { yPercent: 100 });
    gsap.to(overlayRef.current, {
      yPercent: 0,
      duration: 0.8,
      ease: "power3.inOut",
      onComplete: () => {
        // Only trigger the actual Next.js navigation AFTER the overlay covers the screen
        router.push(href); [cite: 4, 15]
      }
    });
  };

  return (
    <TransitionContext.Provider value={{ navigate }}>
      {/* The DOM overlay used for wiping the screen */}
      <div 
        ref={overlayRef} 
        className="fixed inset-0 z-50 bg-black pointer-events-none translate-y-full" 
      />
      {children}
    </TransitionContext.Provider>
  );
};
```

**Code Example: Custom TransitionLink Component**

```tsx
// components/TransitionLink.tsx
"use client";

import React, { useContext } from "react";
import Link, { LinkProps } from "next/link";
import { TransitionContext } from "@/context/TransitionContext";

interface TransitionLinkProps extends React.PropsWithChildren<LinkProps> {
  href: string;
  className?: string;
}

export const TransitionLink = ({ children, href, className, ...props }: TransitionLinkProps) => {
  const { navigate } = useContext(TransitionContext);

  const handleClick = (e: React.MouseEvent<HTMLAnchorElement, MouseEvent>) => {
    e.preventDefault(); // Prevent instantaneous native routing [cite: 3]
    navigate(href.toString());
  };

  return (
    // We still use Next/Link for prefetching, but intercept the click
    <Link href={href} onClick={handleClick} className={className} {...props}>
      {children}
    </Link>
  );
};
```

### Decision Tree: Navigation Implementations

| Use Case | Recommended Component | Rationale |
| :--- | :--- | :--- |
| **Internal Routing with Page Exit Effects** | `<TransitionLink href="...">` | Intercepts click, triggers GSAP timeline, waits for completion, then executes `router.push()` [cite: 3, 4]. |
| **External Links or Anchor Links (`#hash`)** | Native HTML `<a>` tag | Transition wipes should not be applied when leaving the domain or scrolling down a single page [cite: 16]. |
| **Instantaneous Internal Navigation** (e.g., Pagination) | Native Next.js `<Link>` | For fast state changes where blocking the user behind a 1-second animation degrades UX. |

### Anti-Rationalization Rules for Route Transitions

**Rule 1: The Native Link Assumption**
*   **The Temptation:** *"I will just use a standard `<Link>` and run a `useEffect` return cleanup function to animate the component out."*
*   **Why it Fails:** When a user clicks a standard Next.js `<Link>`, the App Router instantly destroys the DOM nodes [cite: 14]. The `useEffect` cleanup function fires instantly, meaning the elements are erased from the browser paint cycle before GSAP can advance the timeline to `progress(0.1)`. The result is a jarring snap to the next page.

**Rule 2: The Framer Motion Deflection**
*   **The Temptation:** *"GSAP route transitions seem overly manual. I will just install Framer Motion and wrap the layout in `<AnimatePresence mode="wait">`."*
*   **Why it Fails:** As of recent versions, `AnimatePresence` suffers from deeply rooted, long-standing architectural bugs within the Next.js App Router context, specifically regarding nested layouts and server component serialization [cite: 13]. Mixing Framer Motion for routing and GSAP for element animation causes severe state desynchronization and bloated bundle sizes.

## 3. Tailwind CSS Coexistence

Tailwind CSS and GSAP utilize fundamentally different paradigms for DOM manipulation. Tailwind generates static CSS classes applied to elements, while GSAP uses JavaScript's `requestAnimationFrame` to calculate mathematical interpolations and applies them as highly specific inline styles (`style="transform: translate3d(...)"`).

The friction between these technologies occurs primarily across two vectors: CSS transitions fighting JS interpolations, and display utilities overriding visibility [cite: 5, 6].

### Technical Reference: Resolving Engine Conflicts

When a DOM element possesses a Tailwind transition class (e.g., `transition-all duration-300`), the browser is instructed by the CSS Object Model (CSSOM) to interpolate any style change over 300 milliseconds. If GSAP simultaneously attempts to animate the `x` position of that same element 60 times per second, the CSS engine intercepts every single GSAP frame update and attempts to apply a 300ms transition to it [cite: 6]. This results in extreme visual lag, stuttering, and CPU thrashing.

**Code Example: The Conflict and the Resolution**

```tsx
// ❌ INCORRECT: Tailwind and GSAP fighting
export function BadComponent() {
  const boxRef = useRef(null);
  
  useGSAP(() => {
    // GSAP tries to move this instantly or over 1s...
    gsap.to(boxRef.current, { x: 200, duration: 1 });
  });

  return (
    // ...but Tailwind forces a 500ms CSS delay on every GSAP inline style update!
    <div ref={boxRef} className="transition-all duration-500 hover:bg-blue-500">
      Lagging Box
    </div>
  );
}

// ✅ CORRECT: Separating concerns
export function GoodComponent() {
  const boxRef = useRef(null);
  
  useGSAP(() => {
    gsap.to(boxRef.current, { 
      x: 200, 
      duration: 1,
      // If we need the element to return to pure CSS control after animating:
      clearProps: "all" 
    });
  });

  return (
    // Target strictly background color for CSS transitions. Leave transform/opacity to GSAP.
    <div ref={boxRef} className="transition-colors duration-500 hover:bg-blue-500">
      Smooth Box
    </div>
  );
}
```

### The `hidden` vs `autoAlpha` Conflict

Tailwind's `hidden` class applies `display: none;`. GSAP animations typically rely on calculating bounding boxes, which return `0` for elements with `display: none` [cite: 17]. Furthermore, GSAP has a specialized property called `autoAlpha`. `autoAlpha: 0` is highly optimized: it sets `opacity: 0` and `visibility: hidden` (preventing pointer events and removing it from the accessibility tree, but retaining its spatial dimensions for layout calculations) [cite: 7, 14].

### Decision Tree: CSS vs GSAP Delegation

| Property Type | Controller | Implementation Strategy |
| :--- | :--- | :--- |
| **Transforms** (`x`, `y`, `scale`, `rotate`) | **GSAP** | Remove Tailwind `transition-all`. Use GSAP `gsap.set()` to establish initial values instead of Tailwind `translate-x-*` classes [cite: 5]. |
| **Visibility / Fade** (`opacity`) | **GSAP** | Use GSAP `autoAlpha` instead of `opacity`. Avoid Tailwind `hidden` class [cite: 7]. |
| **Hover States / Colors** (`bg`, `text`) | **Tailwind** | Use Tailwind `transition-colors hover:bg-gray-200`. This executes efficiently on the GPU without Javascript overhead. |
| **Responsive Layouts** (`flex`, `grid`) | **Tailwind** | Let CSS dictate flow. Use GSAP `Flip` plugin if layout changes must be animated. |

### Anti-Rationalization Rules for Tailwind Coexistence

**Rule 1: The "transition-all" Laziness**
*   **The Temptation:** *"I'm going to apply `transition-all ease-in-out` to my cards so the hover effects look nice, and then use GSAP to animate those cards entering the screen."*
*   **Why it Fails:** `transition-all` affects the `transform` property [cite: 6, 18]. When GSAP fires `y: 50`, the browser blocks the instant layout update and drags it over the transition duration, completely ruining GSAP's easing logic, staggering, and timing mechanisms [cite: 6].

**Rule 2: The `display: none` Trap**
*   **The Temptation:** *"I'll add Tailwind's `hidden` class to my element so it doesn't show up on load, then animate it in with GSAP using `opacity: 1`."*
*   **Why it Fails:** GSAP cannot animate an element out of `display: none` seamlessly if it calculates `opacity`. The element will violently snap into the document flow, disrupting layout constraints. The correct approach is using `visibility: hidden` (via custom CSS) and animating via `autoAlpha: 1` [cite: 7, 17, 19].

## 4. Flash of Unstyled Content (FOUC) Prevention in Server-Side Rendering

FOUC is an exceptionally prominent issue in SSR frameworks like Next.js. The lifecycle of a Next.js page load is:
1.  The server generates the HTML and CSS and sends it to the browser.
2.  The browser immediately paints the HTML using the CSS (First Contentful Paint).
3.  The JavaScript bundle downloads.
4.  React hydrates the application.
5.  `useLayoutEffect` / `useGSAP` fires, allowing GSAP to apply its initial styles.

If GSAP is responsible for an entrance animation (e.g., sliding a hero heading from `opacity: 0` to `opacity: 1`), there is a critical delta between Step 2 and Step 5. During this time, the element renders fully visible at `opacity: 1`. When GSAP initializes in Step 5, it instantly forces the element to `opacity: 0`, resulting in a glaring, ugly flicker—the FOUC [cite: 7, 8].

### Technical Reference: Pure CSS Hiding with `autoAlpha` Initialization

To absolutely guarantee the prevention of FOUC, the initial hidden state **must be defined in CSS, not JavaScript** [cite: 17]. By applying `visibility: hidden` directly via a CSS module or inline style, the browser will never paint the element during Step 2. When GSAP loads, `autoAlpha: 1` will seamlessly restore both visibility and opacity [cite: 7, 19].

**Code Example: Bulletproof FOUC Prevention Strategy**

```tsx
// components/HeroAnimation.tsx
"use client";

import React, { useRef } from "react";
import { useGSAP } from "@gsap/react";
import { gsap } from "gsap";
import { SplitText } from "gsap/SplitText";

gsap.registerPlugin(SplitText);

export default function HeroAnimation() {
  const containerRef = useRef<HTMLDivElement>(null);
  const textRef = useRef<HTMLHeadingElement>(null);

  useGSAP(() => {
    // At this point, the text is completely invisible due to inline CSS.
    // We split the text, prepare the spans, and then animate them.
    const split = new SplitText(textRef.current, { type: "chars" });
    
    // autoAlpha toggles visibility from 'hidden' to 'visible' 
    // AND animates opacity from 0 to 1 simultaneously. [cite: 7, 19]
    gsap.fromTo(split.chars, 
      { autoAlpha: 0, y: 50 }, 
      { autoAlpha: 1, y: 0, stagger: 0.05, duration: 1, ease: "back.out(1.7)" }
    );

    // Make the parent container visible in case we hid the whole block
    gsap.set(containerRef.current, { autoAlpha: 1 });
  }, { scope: containerRef });

  return (
    // CRITICAL: We set visibility: hidden natively in the HTML/CSS payload [cite: 7, 8]
    // Do NOT use opacity: 0, as the invisible text will still block pointer events.
    <div ref={containerRef} style={{ visibility: "hidden" }}>
      <h1 ref={textRef}>
        Immersive GSAP Experience
      </h1>
    </div>
  );
}
```

### Decision Tree: FOUC Management

| Scenario | Anti-FOUC Implementation | Explanation |
| :--- | :--- | :--- |
| **Above-the-Fold Animation** (e.g., Hero Header) | Add `style={{ visibility: 'hidden' }}` to element. Animate with `autoAlpha: 1` [cite: 7]. | Prevents the element from flashing visible during SSR hydration [cite: 8, 19]. |
| **Below-the-Fold Animation** (e.g., ScrollTriggered sections) | No CSS prevention required. Use standard `gsap.from(el, { opacity: 0 })` or `gsap.set(el, { autoAlpha: 0 })` [cite: 20]. | The element is not currently visible to the user during initial hydration, so the FOUC flicker happens off-screen harmlessly [cite: 20]. |
| **No-JS Fallback Requirements** | Add `<noscript>` tag that overrides `.hidden` classes. | If progressive enhancement is strictly required, ensure users without JS enabled can still see the layout [cite: 20]. |

### Anti-Rationalization Rules for FOUC

**Rule 1: The JS Initialization Fallacy**
*   **The Temptation:** *"I want to avoid FOUC, so I will add `gsap.set('.hero', { autoAlpha: 0 })` at the very top of my component, before my animation timeline runs."*
*   **Why it Fails:** `gsap.set()` relies entirely on JavaScript [cite: 8]. Because Next.js serves fully formed HTML to the browser *before* React executes the JS bundle, the user's browser paints the screen exactly as the HTML dictates. The flash occurs before `gsap.set()` is ever parsed by the V8 engine [cite: 8, 17].

**Rule 2: The Opacity Fallacy**
*   **The Temptation:** *"I will use `opacity: 0` in Tailwind (`opacity-0`) to hide the element, then animate it to `opacity-100`."*
*   **Why it Fails:** While this does hide the element visually, `opacity: 0` retains the element's pointer events. If it's a button or a link overlaying the screen, a user can inadvertently click it while it is invisible. `visibility: hidden` (paired with GSAP's `autoAlpha`) properly removes the element from interactivity until the animation dictates otherwise [cite: 7, 14].

## 5. Dynamic Imports and Resource Management for GSAP Payloads

GSAP is a highly optimized library, but robust applications often employ multiple plugins (ScrollTrigger, ScrollSmoother, SplitText, Flip, MorphSVG). Combined with complex React components—such as Three.js canvases (`react-three-fiber`) or heavy SVG illustrations—the initial client-side JavaScript payload can expand significantly, degrading Time to Interactive (TTI) and Largest Contentful Paint (LCP) [cite: 10, 21].

Furthermore, some GSAP setups specifically calculate `window` dimensions or `document` nodes on initialization. If these are processed on the server during SSR, they will trigger fatal hydration mismatch errors (`window is not defined`).

### Technical Reference: `next/dynamic` with `ssr: false`

The solution to both payload bloat and hydration errors is strategic Lazy Loading using `next/dynamic` [cite: 10, 21, 22]. By importing heavy animation modules with the `{ ssr: false }` flag, Next.js explicitly removes the component from the server-rendered HTML payload. The browser only requests and processes the component's JavaScript after the initial page has loaded and hydrated.

**Code Example: Dynamic Imports for Heavy Animations**

```tsx
// app/page.tsx
import React from "react";
import dynamic from "next/dynamic";

// Standard Client Component loaded normally (SSR'd)
import HeroSection from "@/components/pages/homepage/HeroSection";

// Heavy GSAP Component dynamically imported [cite: 21]
// ssr: false ensures the server ignores this file, preventing 'window' errors 
// and deferring the JS payload. [cite: 10, 22]
const DynamicTestimonials = dynamic(
  () => import("@/components/pages/homepage/TestimonialsSection"),
  { 
    ssr: false, 
    loading: () => <div className="min-h-screen animate-pulse bg-gray-100" /> 
  }
);

export default function HomePage() {
  return (
    <main>
      {/* Renders immediately in the HTML payload */}
      <HeroSection />

      {/* Renders only on the client after hydration, preserving initial load speed */}
      <DynamicTestimonials />
    </main>
  );
}
```

**Code Example: Centralizing GSAP Registration**

To avoid redundant imports and prevent multiple instances of ScrollTrigger from conflicting across dynamically loaded boundaries, GSAP initialization should be centralized into a single registry file [cite: 21, 23].

```tsx
// lib/gsapConfig.ts
"use client";

import { gsap } from "gsap";
import { ScrollTrigger } from "gsap/ScrollTrigger";
import { useGSAP } from "@gsap/react";

// Ensure registration only happens once, and only in browser environments [cite: 23]
if (typeof window !== "undefined" && !gsap.core.globals()["ScrollTrigger"]) {
  gsap.registerPlugin(ScrollTrigger, useGSAP);
}

// Export the registered instances to be used application-wide [cite: 21]
export { gsap, ScrollTrigger, useGSAP };
```

### Decision Tree: Import Strategies

| Component Characteristic | Import Method | Justification |
| :--- | :--- | :--- |
| **Above-the-Fold Critical Animations** (e.g., Navbars, Hero text) | Standard `import` | Delaying critical UI causes layout shifts and poor visual perception of speed. Must be included in the initial SSR payload. |
| **Below-the-Fold Complex Animations** (e.g., Heavy ScrollTriggered Data Viz) | `next/dynamic` (No SSR flag) | Defers JS loading but still allows the server to pre-render the HTML shell [cite: 10]. |
| **WebGL / Canvas / Window-Dependent Plugins** (e.g., Three.js, certain Draggable instances) | `next/dynamic` with `{ ssr: false }` | Bypasses server processing entirely, guaranteeing no hydration mismatches for logic heavily reliant on the `window` object [cite: 22, 24, 25]. |

### Anti-Rationalization Rules for Dynamic Imports

**Rule 1: The Server Component Dynamic Import Error**
*   **The Temptation:** *"I want to defer the loading of my data-fetching Server Component, so I will wrap it in `dynamic(() => import(...), { ssr: false })`."*
*   **Why it Fails:** The `{ ssr: false }` configuration is strictly prohibited for Server Components [cite: 10]. Server Components are natively code-split by the App Router framework. Attempting to suppress SSR on a component that exists specifically to run on the server will trigger a compilation error. `ssr: false` is exclusively for Client Components [cite: 10].

**Rule 2: The Over-Lazy Loading Anti-Pattern**
*   **The Temptation:** *"To achieve a 100/100 Lighthouse score, I will dynamically import every single component that uses GSAP with `{ ssr: false }`."*
*   **Why it Fails:** Aggressively applying `{ ssr: false }` to above-the-fold content results in a completely blank page for the user until JavaScript finishes parsing. While Lighthouse might interpret a minimal initial DOM favorably, real-world UX degrades sharply. Search engine crawlers may fail to index content dynamically loaded strictly on the client. `ssr: false` must be reserved for non-critical, below-the-fold, or heavily interactive specific sections [cite: 21].

## Conclusion

Combining the raw imperative animation power of GSAP with the declarative, server-first architecture of the Next.js App Router requires deliberate architectural forethought. By rigorously defining `"use client"` boundaries as wrapper components [cite: 1, 2], developers preserve the SEO and performance advantages of React Server Components. Mitigating route transition issues via customized `TransitionProvider` patterns [cite: 3, 4] resolves the lifecycle limitations inherent to modern Next.js routing.

Furthermore, strictly managing the interface between CSS rendering engines (Tailwind) and JavaScript interpolators (GSAP) eliminates performance bottlenecks [cite: 5, 6]. Implementing CSS-first visibility controls prevents the dreaded Flash of Unstyled Content [cite: 7, 8], and intelligently partitioning payload mass via `next/dynamic` imports ensures that high-fidelity animations do not compromise critical rendering paths [cite: 10, 21]. Adherence to these proven models, and active avoidance of their respective anti-rationalization pitfalls, ensures a flawless synthesis of smooth animations and modern web performance.
